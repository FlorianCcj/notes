Git - For beginner¶

Presentation¶

Ceci n’a pas vocation a etre un tuto, mais a donner les grandes lignes. Cette partie etant celle que je maitrise un peu plus et etant transverse aux differentes technos cette partie va etre un peu plus longue.

Git va gerer votre code grace aux modifications, il ne va pas enregistrer votre code, uniquement les modifications effectuees au projet.

Cette gestion des modifications facilite la gestion des versions, le retour aux versions precedentes ainsi que la revue de code.

Git va gerer votre code en 5 zones:

  1. stash: une zone de sauvegarde temporaire que nous approfondirons un peu plus tard

  2. workspace: la ou est indexe toutes vos modifications

  3. staging: la ou va etre indexe les modifications pretes a etre sauvegardees

  4. local repository: la ou est sauvegarde toutes les modifications

  5. upstream repository: generalement un server distant dans lequel on envoie toutes les modifications pour les sauvegarder, les publier quand on travaille a plusieurs, …

Chaque modification est stockee et referencee par un SHA-1 (ex: 5aaec3b2a305cc47af10e1fd35dfb9bc569cdc1c)

Les references peuvent etre appelees sous plusieurs formes:

  • <nom de branche> : alias pour le commit le plus récent de la branche

  • <tag> : alias de branche, jamais déplacé

  • <SHA-1> : si non ambigu, le début du code suffit

  • <nom>^ : père du commit

  • <nom>^n : nème parent (si parents multiples)

  • <nom>~ : ancètre du dernier commit

  • <nom>~n : nème ancêtre (HEAD~2=HEAD^^)

Utilisation basique¶

TLDR¶

  • git clone <l url que vous aurez sur la page du projet>: rapatrie le repository distant en local

  • git add <le nom du fichier>: cette commande passera le fichier du workspace vers le staging

  • git commit -m "<un message traduisant les modifications misesdans le staging>": fait passer le staging en local repository

  • git push origin master: permet de passer de local a distant

  • git pull: rapatrie les modifications du server au local

  • git status: permet de voir les differentes area

  • git diff: affichera les differences au sein du fichier

  • git diff <source branch> <target branch>: permet de comprendre la difference entre les deux branches

  • git checkout <branch name>: place l environnement de travail sur la branch

  • git checkout <ref>: accede a n importe quel reference

  • git checkout -: revient a la reference precedente

  • git merge <branch de integre dans la branch courante>: pour ajouter les modifications de la branch ma-premiere-feature dans la branch master

  • git merge --ff <branch de integre dans la branch courante>: fera comme precedemment mais ne creara pas de commit de merge

Pour un nouveau projet¶

  • Creer un compte sur un serveur.

  • creer un repository (ou depot en francais)

Rapatrier le depot sur votre ordinateur¶

  • git clone <l url que vous aurez sur la page du projet>: rapatrie le repository distant en local

  • creer un nouveau fichier par exemple README.md

  • ce fichier sera dans le workspace

  • git add <le nom du fichier> ici git add README.md: cette commande passera le fichier du workspace vers le staging et dira a git de suivre les modifications du fichier

  • git commit -m "<un message traduisant les modifications misesdans le staging>" par exemple git commit -m "begin documentation": fait passer le staging en local repository

  • git push origin master: permet de passer de local a distant

Premiere edition¶

  • editer votre fichier

  • git status: permet de voir les differentes area

  • git diff: affichera les differentes modifications au sein des fichiers

  • git add <le nom du fichier>

  • git commit -m "<un message traduisant les modifications misesdans le staging>" par exemple git commit -m "begin documentation": fait passer le staging en local repository

  • git push origin master: permet de passer de local a distant (le push n est pas necessaire a chaque commit, vous pouvez le faire toutes les heures, demi journee, jour (maximum))

Debut des branches¶

  • Les branches reviennent un peu a faire copier coller du dossier puis de modifier le projet, mais tout en gardant un lien avec le dossier d origin

  • vous avez deja rencontrer une branche dans le paragraphe precedent git push origin master, master est le nom de la branche

  • pour travailler il est recommande de ne jamais modifier directement sur la branche master

  • git branch ma-premiere-feature: va creer la branch ma-premiere-feature

  • git checkout ma-premiere-feature: place l environnement de travail sur la branch ma-premiere-feature

  • les deux precedentes commandes peuvent etre resume par git checkout -b ma-premiere-feature

  • tous les commit que vous ferez a partir de ce moment seront reference comme faisant partie de la branch ma-premiere-feature

  • git checkout master: replace l environnement de travail sur la branch master

    • il faut savoir que git checkout permet d acceder a n importe quel reference: un tag, une branch, un commit, …

    • de plus vous pouvez facilement revenir sur la reference precedente avec git checkout -

  • git pull si master a ete modifier sur le depot distant cela permet de rapatrier les modifications sur votre branch master local

  • git diff <source branch> <target branch>: permet de comprendre la difference entre les deux branch

  • git diff ma-premiere-feature master

  • git merge ma-premiere-feature: pour ajouter les modifications de la branch ma-premiere-feature dans la branch master

  • ou git merge --ff <branch de integre dans la branch courante>: fera comme precedemment mais ne creara pas de commit de merge

  • git push origin master: permet d envoyer les modifications de la branch sur le depot distant

  • l’idée sera a l avenir de dire a l un de vos collegue “j ai travailler sur la branch ma-premiere-feature peux tu la recuperer la verifier et si tout te conviens alors on integrera les modifications dans master”

  • aujourd hui vous travaillez seul ? faites la meme chose “je verifie que j ai bien code, que je n ai pas laisse de commentaires inutiles, etc”

Un petit deuxieme pas¶

TLDR¶

  • proposition de nommage de commit <type>(<scope>): <subject>

  • git add .: a proscrire

  • git add -p: recommander, permet de commit portion par portion

  • git diff --staged: affiche le diff dans la staging area

  • git diff --color-words: retire tout les espaces de l affichage et affiche le differenciel mot par mot

  • git diff --color-words=.: utilise les regex, du coup affiche le differenciel charactere par charactere

  • git rebase <nom de branch>: recupere les commit de retard par rapport a la branch d ou est issue votre branch

  • git stash: sauvegarde temporairement

  • git stash list: affichera les differents stash sauvegardees

  • git show stash@{0}: affichera les modifications enregistrees dans le stash

  • git stash drop stash@{0}: supprimera le dernier stash

  • git stash pop: supprimera le stash@{0} et re-appliquera les modifications enregistrees

Bien nommer un commit¶

Tout d abord il faut bien conciderer que chaque commit doit sauvegarder un seul sujet, il ne faut pas avoir un commit creation des todos, management des listes, gestions des utilisateurs. Ce commit devrait etre separer en au moins 3 commit.

Pour un bon nommage de commit je recommande la convention que google a mis en place pour angular

Pour resumer: * <type>(<scope>): <subject> * type: la grande thematique du commit: build, ci, docs, feat, fix, perf, refactor, style, test (non exclusif mais je pense que cela couvre une bonne majorite tout de meme) * scope: plus metier, dans l exemple precedent cela peut etre todo, list, user * subject: la description precise de ce qui a ete modifie (generalement ne cite pas le nom des fichiers editees)

Meme si ceci n est qu une proposition habituez-vous a bien separer vos commit et a bien les nommer.

La staging area¶

  • tres bien on doit separer les commit et bien les nommer mais git add <nom du fichier ne le permet pas si j’ai plusieurs fonctionnalites dans le meme fichier.

  • git add <nom du fichier>: permet effectivement d ajouter tout un fichier, vous pouvez egalement ajouter une liste de fichier voir un dossier, tant qu il concerne bien une seule et meme fonctionnalitee

  • git add .: celui la est vu et revu dans de nombreux tutos, je vous recommande de ne l utiliser qu avec d immense precaution … voir pas du tout

  • git add -p: mon chouchou, permet d ajouter portion de code par portion de code, git vous demandera de choisir

    • y: stage this hunk

    • n: do not stage this hunk

    • q: quit; do not stage this hunk nor any of the remaining ones

    • a: stage this hunk and all later hunks in the file

    • d: do not stage this hunk nor any of the later hunks in the file

    • /: search for a hunk matching the given regex

    • s: split the current hunk into smaller hunks

    • e: manually edit the current hunk

    • ?: print help

    • les 4 options que j utilise principalement sont y, n, s et e

  • git add -u: ajoute les modifications des fichiers DEJA SUIVIS, les fichiers qui ne l etaient pas encore ne seront pas prit en compte

  • git commit -a -m "mon super message" revient a faire une git add -u puis un git commit -m "mon super message"

J ai des lignes en trop¶

Parfois le git diff affiche des lignes ajoutees et supprimees … mais c est quoi ? ce sont des ajouts et des retraits d espaces ou de tabulations ca peut etre ennuyeux. Voici quelques commandes supplementaires

  • git diff --staged: affiche le diff dans la staging area

  • git diff --color-words: retire tout les espaces de l affichage et affiche le diffrenciel mot par mot

  • git diff --color-words=.: utilise les regex, du coup affiche le differenciel charactere par charactere

Recuperation de modification¶

  • c est bien beau tout ca mais moi je travaillais sur la branch ma-deuxieme-feature comment je recupere tes modifications que tu as merge dans master, je fais un git merge ?

  • du tout, cela va creer un commit de merge qui n a rien a faire la

  • git checkout master

  • git pull pour rapatrier l ensemble des modifications

  • git checkout ma-deuxieme-feature

  • git rebase master: va permettre de recuperer sur votre branch l ensemble de commit qui aura ete envoye sur master (ou le nom de votre branch source si ce n est pas master)

  • vous aurez desormais une branch a jour en local

  • Attention au prochain push que vous allez faire, il sera necessaire de le forcer avec un -f car vous aller devoir ecraser l’historique de cette branche

J ai fait des modifications mais je n ai pas encore commit … mais je dois recuperer les mises a jour de master¶

Le stash est fait pour vous, ca revient un peu a un commit temporaire

  • git status: faite ca avant et apre le git stash ce sera plus parlant

  • git stash: va creer un commit temporaire de reference stash@{0} et va retirer les modifications effectuees dans les fichiers suivi dans la working area

  • git stash list: affichera les differents stash sauvegardees

  • git show stash@{0}: affichera les modifications enregistrees dans le stash

  • git stash drop stash@{0}: supprimera le stash et decalera tout, l ancien stash@{1} deviendra le stash@{0} etc

  • git stash pop: supprimera le stash@{0} et re-appliquera les modifications enregistrees

Utilisation avancee¶

Les points touchy et les points moins urgents a voir.

TLDR¶

  • git commit --amend: va prendre ce que vous avez mit dans la staging area pour l ajouter dans le precedent commit

  • git reset <nom du fichier dans la staging area>: permet de passer un fichier de la staging area vers la working area

  • git reset HEAD: va effectuer la command precedente pour l ensemble de la staging area

  • git reset HEAD^: va supprimer le dernier commit

  • git reset HEAD^^: va supprimer les 2 derniers commit

  • git reset HEAD~25: va supprimer les 25 derniers commit

  • git rebase -i <branch source> ou git rebase -i HEAD~10: permet de re-aranger les commit

  • git show <ref de commit> pour voir les commits

  • git fetch rapatrie les modifications sans ecraser votre branch

  • git blame <nom du fichier> permet d afficher le nom du dernier modificateur des differentes lignes du fichiers

  • git log: affichera les differents derniers commit

  • git log -1: affichera uniquement LE dernier commit

  • git log --oneline: affichera les commit mais en simplifier

  • git log --graph --oneline --decorate: avec plusieur branch cela permet d avoir une visualisation d ensemble

J ai oublier une portion de code dans mon commit comment je fais¶

  • le moyen hardcore mais tout est a refaire

    • git stash: va sauvegarder les modifications qui n ont rien a voir avec le commit

    • git reset HEAD^: va supprimer le commit et vous re-affichera les modifications dans la working area

    • git add <vos fichiers de-commiter>: les remettra dans la staging area

    • git stash pop: re-affichera les midufications stash

    • git add <ce que vous aviez oublie>

    • git commit -m <votre message>

  • le moyen soft

    • git add <vos fichiers oublies>

    • git commit --amend: va prendre ce que vous avez mit dans la staging area pour l ajouter dans le precedent commit

  • je profite de ce paragraph pour presenter reset

    • git reset <nom du fichier dans la staging area>: permet de passer un fichier de la staging area vers la working area

    • git reset HEAD: va effectuer la command precedente pour l ensemble de la staging area

    • git reset --hard <nom du fichier dans la staging area>: supprime la modification du fichier present dans la staging area

    • git checkout <nom du fichier dans la working area>: supprime la modification du fichier present dans la working area

    • git reset HEAD^: va supprimer le dernier commit

    • git reset HEAD^^: va supprimer les 2 derniers commit

    • git reset HEAD~25: va supprimer les 25 derniers commit

    • git reset --mixed: comportement par default, revient a faire git reset HEAD

    • git reset --soft: supprimera les modifications ciblées (commit) et les mettra dans la staging zone

    • git reset --hard: supprimera definitement les modifications ciblées.

Bon rearangeons nos commit¶

Plusieurs problemes se posent avec ce qu on a vu

  • git commit -m "list todo"

  • git commit -m "fix list"

  • git commit -m "fix list"

  • git commit -m "fix list"

  • git commit -m "fix list"

  • 15 commit sur le sujet des list

  • les commits de fix qui n en finissent plus

  • ramener sur la branch master ca donnerait 50% de commit de fix 49% de commit de merge … pas tres lisible

  • une solution serait a git reset HEAD~<le nombre de commit fait dans la branche> ?

  • c est moche mais oui ca marcherait, il ne restera plus qu a tout recommit sujet par sujet

  • Arrivé du git rebase

  • Mais qu est ce qu il raconte … git rebase ca permet juste de ratraper le retard de la branch source ?

  • je vous presente git rebase -i <branch source> ou git rebase -i HEAD~10

  • cette commande vous affichera l ensemble des commit de votre branch et vous permettra de les reorganiser, les editer, les renommers etc

  • generalement vous aurez les explications de chaque command dans le texte qui s affichera. Lisez bien l integralite.

  • c est tout ce que tu as a dire ? utilisez git rebase -i ? c est pas terrible

  • quatre choses

    • d abord si vous ne connaissez pas les commandes vous ne pourez pas les utilisez, il fallait bien au moins vous l introduire

    • apres un git rebase -i tout l historique est boulverse, il faudra un git push -f pour l imposer au server

    • attention si vous reorganiser les commit alors que vous avez modifier deux fois la meme portion de code git rebase -i va vous demandez beaucoup de modification de commit, ne prenez pas peur, lisez ce que git vous dit et corrigez tranquillement vos fichiers

    • utilisez git show <ref de commit> pour voir les commits, meme si vous les avez bien nommees, revoir ce que vous avez fait dedans permettra de vous rassurer

Verifiez avant de pull¶

Le git pull va rapatrier les modifications du serveur distant et ecraser votre branch local. En realite on peut rapatrier les references sans ecraser la notre.

par exemple sur votre branch master,

  • git checkout master

  • #git pull ecraserait votre branch

  • git fetch rapatrie les modifications sans ecraser votre branch

  • git diff master origin/master: vous pourrez alors demander a git la difference entre le distant et le local

Des supers fichiers¶

.gitconfig¶

c est dans ce fichier que vous trouverez vos configurations, vos alias, vos parametres d authentifications (sous linux generalement il est dans l home de l utilisateur ~/.gitconfig)

.gitignore¶

Quand vous builder, quand vous rapatriez des library externe, ou autre, vous creer des fichiers qui n ont rien a faire dans git.

Grace a ce fichier .gitignore vous pourrez les lister pour bien preciser a git de ne jamais les prendre en compte.

.gitkeep¶

Dans certaine situation nos scripts ont besoin d un dossier pour stocker des fichiers generes ou des fichiers uploades, malheureusement git ne prend pas en compte les dossiers vides.

On creer donc un fichier vide dans ces dossiers a concerver pour faire comprendre aussi bien a git qu au autre contributeur de ne pas supprimer ce dossier. Par convention on appelle ces fichiers .gitkeep

Qui qu a fait quoi?¶

Quand vous travaillez seul ce n est pas tres important. Mais quand vous travaillez a plusieurs, parfois vous ne comprenez pas certain code … bon outre le fait que si ce code est trop complique il faudrait peut etre le refactorer, pour le comprendre il suffit de demander a son autheur de l expliquer.

git blame <nom du fichier> permet d afficher le nom du dernier modificateur des differentes lignes du fichiers

Qu est ce qui c est passe?¶

Pour ceux qui ont regarder des tutos, vous l aurez vite vu, moi je l aborde que maintenant.

  • git log: affichera les differents derniers commit

  • git log -1: affichera uniquement LE dernier commit

  • git log --oneline: affichera les commit mais en simplifier

  • git log --graph --oneline --decorate: avec plusieur branch cela permet d avoir une visualisation d ensemble

Que et Quand verifier¶

  • Avant de commit toujours verifier ce que vous allez commit avec git diff --staged

  • avant de recuperer un stash, toujours verifier le continue avec git show stash@{0}

  • avant de merge toujours verifier que l on a bien recuperer la branch source, sur la branch en cours de developpement faire git rebase <branch source>

  • avant de merge, toujours verifier le diff entre les deux branch git diff <branch source> <branch target>

  • en cours de git rebase -i, toujours verifier les commit que l on va manipuler avec git show <numero du commit>
